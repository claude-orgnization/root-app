# カスタム開発エージェント フロー設計書

## 概要

業務開発の一連の流れを Claude Code エージェントが実行するためのフロー定義。
各フェーズをスキル（slash command）として分離し、オーケストレーター・エージェントが全体を制御する。

## フロー全体図

```
┌─────────────────────────────────────────────────────┐
│                /dev-flow (オーケストレーター)          │
│                                                     │
│  ① /spec-define ─── 要件仕様の定義                   │
│       ↓                                             │
│  ② /arch-design ─── データアーキテクチャ設計          │
│       ↓                                             │
│  ③ /tech-stack ──── 技術スタック決定                  │
│       ↓                                             │
│  ④ /task-breakdown ─ タスク分解                      │
│       ↓                                             │
│  ⑤ /tdd-cycle ───── テスト作成 → 実装 → リファクタ    │
│       │              (RED → GREEN → REFACTOR)        │
│       │              ループ実行                       │
│       ↓                                             │
│  ⑥ /e2e-test ────── E2Eテスト（必要時）              │
│       ↓                                             │
│  ⑦ /create-pr ───── プルリクエスト作成               │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 各フェーズの詳細

### Phase 1: 要件仕様定義 (`/spec-define`)
- **入力**: ユーザーからの機能要望・課題説明
- **出力**: `docs/SPEC.md` の更新または新規作成
- **処理内容**:
  - ユーザーへのヒアリング（不明点の質問）
  - 機能要件・非機能要件の整理
  - ユースケース・画面フローの定義
  - 受け入れ基準（Acceptance Criteria）の明記

### Phase 2: データアーキテクチャ設計 (`/arch-design`)
- **入力**: 確定した仕様書
- **出力**: `docs/ARCHITECTURE.md` の更新
- **処理内容**:
  - データモデル / 型定義の設計
  - API エンドポイント設計
  - コンポーネント構成・状態管理方針
  - ディレクトリ構成の決定

### Phase 3: 技術スタック決定 (`/tech-stack`)
- **入力**: アーキテクチャ設計
- **出力**: `docs/TECH-STACK.md` または `CLAUDE.md` の技術スタックセクション更新
- **処理内容**:
  - 既存スタックとの整合性確認
  - 新規ライブラリの選定理由の記録
  - `package.json` への依存追加（必要時）

### Phase 4: タスク分解 (`/task-breakdown`)
- **入力**: 仕様書 + アーキテクチャ
- **出力**: GitHub Issues or TodoList
- **処理内容**:
  - 実装タスクを粒度の細かい単位に分解
  - 各タスクの依存関係を明示
  - 優先順位付け
  - 見積もり（S/M/L）

### Phase 5: TDDサイクル (`/tdd-cycle`)
- **入力**: タスクリスト + 仕様
- **出力**: テストコード + 実装コード + リファクタ済みコード
- **処理内容**:
  1. **RED**: 失敗するテストを書く
  2. **GREEN**: テストを通す最小実装
  3. **REFACTOR**: コード品質を改善
  4. 次のタスクへ → 1に戻る
- **繰り返し条件**: 全タスク完了まで

### Phase 6: E2Eテスト (`/e2e-test`)
- **入力**: 実装済みコード
- **出力**: E2Eテストコード + テスト結果
- **処理内容**:
  - 主要ユーザーフローのE2Eテスト作成
  - テスト実行・結果確認
  - 必要に応じて修正

### Phase 7: プルリクエスト作成 (`/create-pr`)
- **入力**: 実装済み・テスト済みコード
- **出力**: GitHub Pull Request
- **処理内容**:
  - 変更差分の確認
  - PR タイトル・説明文の自動生成
  - レビュアー指定（設定があれば）
  - CI チェックの確認

## エージェント間の連携

```
dev-flow (orchestrator)
├── spec-agent      ── 仕様策定の専門エージェント
├── architect-agent ── 設計の専門エージェント
├── tdd-agent       ── TDDサイクルの実行エージェント
└── review-agent    ── コードレビュー・PR作成エージェント
```

## 使い方

```bash
# 全フローを実行
/dev-flow "ユーザー認証機能を追加したい"

# 個別フェーズを実行
/spec-define "ログイン機能の要件を定義して"
/tdd-cycle "src/hooks/useAuth.ts のテストと実装"
/create-pr
```
